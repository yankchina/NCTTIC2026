### 3. 数据库 Schema 设计（迭代过程）

经过多轮讨论，最终确定以下设计要点：

#### 获奖等级双字段设计
- `award_level` INTEGER (1–4)：用于排序与范围筛选，数值越小等级越高
- `award_title` ENUM：用于界面展示，值为（特等奖/一等奖/二等奖/三等奖/优秀奖/专项奖）
- 背景：不同赛事、不同地区对同一等级的称谓不同，两字段解耦

#### 双快照策略（获奖记录中的学校信息）
- 保留 `school_id` 外键（用于关联查询与统计）
- 同时快照 `school_name_snapshot` 和 `school_province_snapshot`
- 背景：教师可能在不同学校间调动，历史获奖记录需固化当时的学校

#### 敏感信息隔离
- `teacher_private` 独立表存储证件号、银行账号
- 主表与敏感表 1:1 关系，应用层按需加载
- 后续规划应用层加密

#### 标签多态设计
- `tags` 表定义标签，含 `is_system` 和 `tag_key` 字段
- `entity_tags` 通过 `entity_type + entity_id` 挂载到任意实体
- 预置系统标签：**收藏**（favorite）、**关注**（follow）
- 收藏页可聚合展示跨实体类型的内容

#### 教师信息扩充
新增字段：性别、出生日期、籍贯、民族、政治面貌、身份证件类型、最高学位、毕业学校、研究方向、个人主页

#### 学校信息扩充
新增字段：英文全称（`name_en`）、中文缩写（`abbr_zh`）、英文缩写（`abbr_en`）、官网（`website`）

## 决策记录

| 编号 | 决策 | 原因 |
|---|---|---|
| D-001 | 先做 macOS 端，再做 iOS | 录入场景在桌面，移动端以查阅为主 |
| D-002 | 使用 PostgreSQL 而非 SQLite | 已有数据在 PostgreSQL，复杂查询支持更好 |
| D-003 | 获奖等级用双字段 | 称谓因赛事而异，展示与逻辑需解耦 |
| D-004 | 教师调动用快照解决 | 外键无法保留历史状态，快照是正确的历史型数据建模方式 |
| D-005 | 标签用多态关联表 | 避免为每个实体单独建标签关联表，收藏页可跨类型聚合 |
